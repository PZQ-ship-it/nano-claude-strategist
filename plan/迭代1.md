🗺️ 迁移迭代 1 详细实施方案 (供您全局把控)
我们在 nano-claude-code 中对应的改造目标非常明确，集中在 task/ 目录下的三个核心层：

数据模型层 (task/types.py)：

找到定义任务实体的地方（通常是 Task dataclass 或 Pydantic 模型）。

直接注入新字段：expected_value, p_success, duration_hours, sunk_cost_hours, deadline_timestamp, dependencies（依赖任务 ID 列表）。

植入数学引擎：在此类中直接实现 calculate_dynamic_score(current_timestamp) 函数，包含基于广义逻辑函数的双曲贴现计算与沉没成本防抢占机制。

存储与状态管理层 (task/store.py)：

引入 networkx：在存储管理器（如 TaskStore）的初始化方法中新增 self.dag = nx.DiGraph()。

生命周期挂载：新增一个私有方法 _sync_dag(self)，在每次加载、新增、更新、删除任务后调用它，清空并重新将内存中的任务映射为节点和有向边。

防死锁强校验：每次有向边发生改变时，执行 nx.is_directed_acyclic_graph，如遇死锁立即抛出 ValueError 阻断修改。

级联衰减机制：新增 cascade_ev_decay 业务方法，利用 nx.descendants 寻找波及节点并削减预期收益。

大模型工具与渲染层 (task/tools.py)：

更新工具 Schema：让创建任务和修改任务的 Tool 支持接收这些新的运筹学参数，并透传给 Store。

重写查询展示逻辑：

利用 store.dag.predecessors 检查是否存在任何状态未完成的前置节点，若有，强制给该任务打上 [BLOCKED] 标签。

对 Ready 的任务实时计算 Score。

严格按 Score 降序排列，向 LLM（以及终端）输出带有 Score、EV、Laxity(h) 等硬核指标的 Markdown 表格。
